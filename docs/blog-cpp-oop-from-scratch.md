# 從「讀完教材還是看不懂」到真正懂 CsvLogger：一次逐行拆解筆記

上一篇寫完 Catch2 的踩雷紀錄之後，我意識到自己完全沒學過 C++ 的物件導向，於是請 AI 幫我從零整理了一份完整的 OOP 教材，從 `struct`/`class` 的差異一路講到繼承、多型。讀完那份教材，我以為自己懂了——結果一回頭看自己在挑戰 0.3 寫的 `CsvLogger`，還是完全看不懂它在幹嘛。這件事本身就值得記一筆：**讀懂抽象規則，跟看得懂一段實際程式碼，是兩種完全不同的能力**，中間那段落差沒人幫你搭橋，自己想破頭也很難跨過去。

## 卡住的地方：不只是 OOP 概念，連檔案本身是什麼都沒搞懂

回頭看我的疑問，其實第一個問題根本還沒進到 OOP——我連「為什麼這個檔案叫 `.hpp`」都沒搞懂。教材裡教了建構子、解構子、封裝，但完全沒解釋 C++ 的檔案為什麼要分成 `.hpp` 跟 `.cpp`，這個空白直接卡死了我後面所有的理解，因為我根本不知道自己在讀的這份 `csv_logger.hpp` 在整個編譯流程裡扮演什麼角色。

## `.hpp` 到底是什麼：先搞懂編譯模型

C++ 的每一個 `.cpp` 檔案是**各自獨立編譯**的，編譯器一次只看得到一個 `.cpp` 檔案的內容（這種獨立編譯的單位叫 translation unit）。如果 `main.cpp` 想用某個 class，編譯器在編譯 `main.cpp` 的當下就必須「看得到」那個 class 長什麼樣子，才能檢查程式碼寫得對不對。

`#include` 做的事情非常單純、甚至有點原始：**在編譯正式開始之前，把指定檔案的完整內容，原封不動地貼到 `#include` 那一行的位置**。純文字複製貼上，沒有更複雜的機制。所以 `#include "csv_logger.hpp"` 效果就等同於手動把整份 `csv_logger.hpp` 貼進那個位置。

傳統上 C++ 專案把程式碼分成兩種檔案：`.cpp`（實作檔，裡面是函式真正的內容，會被單獨編譯成目的檔）和 `.h`/`.hpp`（標頭檔，主要放宣告——告訴其他檔案「這個東西存在、長什麼樣子」）。`.hpp` 是社群慣例，用來標示「這是 C++ 專用的標頭（用了 class 這種 C 沒有的語法）」。

那為什麼 `CsvLogger` 沒有照傳統拆成 `.hpp`（宣告）+ `.cpp`（實作）兩個檔案，而是全部塞進一個 `.hpp` 裡？因為它的每個成員函式，函式主體都直接寫在 class 定義**裡面**，這種寫法叫 **inline 定義**。C++ 規定：函式定義只要寫在 class 內部就自動被當成 `inline`，代表就算這份 `.hpp` 被好幾個不同的 `.cpp` 各自 `#include`（等於同一段程式碼在最後連結時出現很多份），連結器也不會抱怨「重複定義」。這是刻意的選擇：對這麼小的 class，拆成兩個檔案只是多一道手續，沒有實質好處；等專案變複雜、`.cpp` 常常單獨修改時，拆檔案才有意義。

至於檔案最開頭的 `#pragma once`，是所謂的 include guard：避免同一份標頭在同一個 `.cpp` 裡（透過不同路徑的 `#include` 鏈）被貼進去兩次，導致「class 被定義兩次」的編譯錯誤。

## 逐行拆解：CsvLogger 真正在做什麼

搞懂檔案本身是什麼之後，回頭拆 class 內容，其實每一行都對得上前面 OOP 教材教過的概念，只是要有人指出「這裡就是那個概念」：

**建構子**裡 `: file_(path)` 是成員初始化列表——`std::ofstream` 的建構子只要給它一個路徑字串就會嘗試開檔，所以「開檔」這個動作實際發生在這一行，不是後面 `{}` 裡任何一行。接著用 `is_open()` 檢查開檔有沒有成功，失敗就 `throw` 例外——與其放任一個「內部檔案根本沒打開」的半殘物件被建出來、之後每次寫入都默默失敗，不如一開始就讓建構直接失敗、清楚報錯。剩下的迴圈就是把 header 字串一個個寫進檔案、逗號分隔、結尾換行，組成合法的 CSV 格式。

**解構子**裡的 `file_.close()` 嚴格說其實可以不寫（`std::ofstream` 自己的解構子本來就會做這件事），寫出來純粹是教學用意：讓「資源在解構時被歸還」這件事在程式碼裡真的看得見，不是藏在標準函式庫看不到的角落。

**`= delete` 複製、`= default` 搬移**這兩組，一路對回教材裡的 Rule of Three / Rule of Five：兩個 `CsvLogger` 共用同一個底層檔案控制代碼會是災難（雙方解構時都想關同一個檔案），乾脆在編譯期直接擋掉複製；但搬移沒有這個問題（所有權整個轉移，永遠只有一個物件真正擁有資源），而且 `std::ofstream` 本身已經正確支援搬移，所以讓編譯器自動生成（`= default`）就夠了，不用自己手刻。

**`row()`** 的邏輯跟建構子裡寫 header 那段幾乎一模一樣，只是這次寫的是一串 `double` 而不是字串。

## 學到的事

這次真正學到的東西，其實不是任何新的 C++ 語法——教材裡該有的都有了。學到的是**抽象規則跟具體程式碼之間需要一座橋**：讀完「建構子負責初始化」「RAII 是建構子拿資源解構子還資源」這些規則式的敘述，跟看著一段真實程式碼、能一眼認出「喔，`: file_(path)` 這行就是規則講的『拿資源』那一步」，中間隔著一段不小的距離。對我來說，光讀抽象教材填不滿這段距離，得回頭拿一份自己寫過、有實際脈絡的程式碼，一行一行對照著問「這行對應到哪條規則」，才真正接得起來。

以後遇到新概念，可能會直接跳過「先讀完整教材」這一步，改成「教材 + 手邊一份具體程式碼同時對照著看」，兩邊來回會比單向讀完再回頭看有效率得多。
